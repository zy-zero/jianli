  • QMS风险管理项目

    主要使用.net core + ABP.Core框架 + postgresql搭建后端，前端使用react搭建；

    项目是解决一些制造商的风险评估和管理，这个项目耦合性极高，几乎每个功能都跟其他地方有耦合，在项目中，主要负责后端业务代码的开发和部分前端开发，解决部分性能问题等。

    项目中excel table中后端缓存加载10多万行，200多万个单元格的数据，会出现性能问题；

    一、查询数据库优化，在项目中.net core + EF core 查询数据库的时候，使用linq语句写得代码会生成很多不必要的子查询，先是优化了一部分这些子查询的linq语句，以及适当分页会减少部分查询时间。

    二、代码问题，项目中使用了比较多的循环语句去组装缓存数据，查询出来的数据全部都是用List这个类去组装的，这里适当的修改为Dictionary类或者是HashSet类，减少List类的使用，会提升较大性能，减少空间复杂度。

    三、适当使用并行计算，部分较大的非异步代码，可以使用并行计算去充分利用CPU的多核心能力，去减少循环里面的耗时（需要注意的是，并行计算里面，不要使用List类这些非线程安全的数据的修改）。

    前端使用handsontalbe做excel table的显示，由于最开始的技术选型，handsontable自身的一些性能限制，前端在合并单元格（merge cell）比较多的时候也有部分性能问题。

    handsontable如果是单纯的渲染每个单元格，然后去简单编辑，是不会产生较大问题的。由于我们项目对每个单元格的渲染都做了自定义，在每次去更新数据或者是触发handsontable自身的重绘逻辑的时候，merge cell带来的性能问题会放大。

    handsontable本身支持虚拟渲染，但是merge cell如果存在超出屏幕的行数，比如渲染30行，但是merge 100行数据，他会将这100行数据都进行渲染，所以导致虚拟dom实际上没有起效果，由于渲染和计算了比较多的单元格（dom），页面会非常卡。另外一个问题是，当我们存在几万行数据大量单元格需要去计算合并的时候，js的性能瓶颈也会出现。
    解决方案是我们只去计算和渲染屏幕第一行到屏幕下方几行的数据的单元格合并，当用户上下滚动的时候，需要重新去计算屏幕内的数据，并重新渲染合并的单元格。另外就是行高的限制，如果不限制行高，merge cell也会出现一些问题，解决方案固定行高，用户可以用按钮去控制最大行高，默认200px。
